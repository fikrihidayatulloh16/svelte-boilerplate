================ boilerplate feature core ===============

Agar Anda bisa langsung tancap gas ke proses bisnis (misal: fitur e-commerce, sistem sekolah, atau B2B), berikut adalah komponen wajib yang harus ada di boilerplate Anda selain gRPC:

1. Authentication Interceptors (The "Heart") ✅
Di gRPC, kita tidak memakai axios.interceptors. Kita memakai Connect Interceptors.
Mengapa wajib? Anda tidak mau menulis kode untuk mengambil token dari localStorage dan menaruhnya di header di setiap fungsi API.

Logic: Otomatis menyisipkan Authorization: Bearer <token> ke setiap request.

Logic: Menangani error 401 (Unauthenticated) secara global untuk memaksa user logout atau refresh token.

2. Global Feedback System (Toasts & Loading Bar)
Jangan biarkan user menebak apakah aplikasi sedang bekerja atau error.

Sonner (Toasts)✅: Integrasikan gRPC error codes ke dalam Toast. Jika backend mengirim error PERMISSION_DENIED, aplikasi otomatis memunculkan pesan merah "Anda tidak punya akses".

NProgress v1.0✅: Indikator loading di bagian atas layar saat navigasi antar halaman atau saat request gRPC sedang berjalan.

3. Form Handling & Validation (The "Muscle")
Bisnis itu 80% adalah mengisi form.

Zod Integration✅: Karena gRPC sudah memberi tipe data, Anda tinggal menyambungkannya dengan Zod untuk validasi di sisi klien.

Svelte-Forms-Lib (jadinya superform) / Superforms✅: Mengelola state input (dirty, touched, error) agar Anda tidak membuat variabel $state manual untuk setiap kolom input.

4. Theme & Layout Engine
Dark Mode ✅: Pasang svelte-themes. Di tahun 2026, dark mode bukan fitur bonus, tapi kewajiban.

Responsive Sidebar/Navbar ✅: Buat komponen shell aplikasi yang sudah siap dengan slot untuk profil user dan menu navigasi.

5. Deployment & Dev-Environment Consistency
Docker & Docker Compose: Karena Anda suka self-hosting di Proxmox, boilerplate Anda harus punya Dockerfile yang sudah dioptimasi untuk SvelteKit (Node.js adapter) agar sekali push, langsung jalan di server.

Environment Manager: Skema .env.example yang jelas untuk memisahkan URL gRPC backend (Local vs Production).

Komponen,Status,Mengapa Penting?
gRPC Mocking,✅ Aman,Bisa ngoding tanpa nunggu backend jadi.
Interceptors,✅ sudah,Agar tidak perlu urus token manual di setiap fitur.
Global Error Handler,✅ Belum,Agar error gRPC tidak membuat aplikasi crash.
Zod Schema,✅ Belum,Validasi input user sebelum dikirim ke backend.

```Protobuf
syntax = "proto3";

package v1;

// Objek data UserLogin
message LoginRequest {
    string email = 1;
    string password = 2;
}

// Kontrak jawaban dari server (Response)
message LoginResponse {
  string session_token = 1;
  User user = 2;
}

message User {
  string id = 1;
  string email = 2;
  string full_name = 3;
  string role = 4;
}

message LogoutRequest {
  // Kosong tidak masalah! 
  // Di gRPC web/SvelteKit, session_token biasanya dikirim lewat 
  // Header (Metadata), bukan lewat body request agar lebih aman.
}

message LogoutResponse {
  bool success = 1;
  string message = 2;
}

// Definisi layanan (Service)
service AuthService {
  rpc Login(LoginRequest) returns (LoginResponse);
  rpc Logout(LogoutRequest) returns (LogoutResponse);
}

```

===================Dark Mode Guide===========================

#############################################################

Prinsip 1: Mentalitas "Pasangan Warna"

Ini adalah kunci utamanya. Jangan pernah mendesain sebuah elemen hanya dengan warna terang.

- Pikirkan: "Jika ini putih di siang hari, apa warnanya di malam hari?"

- Praktik: Setiap kali Anda mengetikkan class warna latar belakang (bg-), warna teks (text-), atau warna batas (border-), segera ketikkan juga pasangan dark:-nya.

Contoh:

- bg-white Pasangannya dark:bg-gray-900

- text-gray-900 Pasangannya dark:text-white

border-gray-200 Pasangannya dark:border-gray-700

############################################################

Prinsip 2: Gunakan "Aksen Utama" yang Fleksibel

Aplikasi Anda pasti punya warna utama (misal: Biru/blue-600).

- Di mode terang, warna terang seperti blue-600 terlihat jelas.

- Di mode gelap, warna yang terlalu pekat kadang "tenggelam".

Tips: Gunakan shade (tingkat kecerahan) yang sedikit berbeda untuk mode gelap.

- Tombol Primer: bg-blue-600 hover:bg-blue-700 dark:bg-blue-500 dark:hover:bg-blue-600

- Teks Tautan (Link): text-blue-600 dark:text-blue-400

#############################################################

Prinsip 3: Pewarnaan Dasar (Base Colors) di Global CSS

Anda tidak perlu (dan tidak boleh) menuliskan warna latar belakang <body> berulang kali di setiap halaman. Biarkan Tailwind menanganinya di level global (file src/routes/layout.css atau app.css).

Kita sudah melakukannya sebelumnya dengan menambahkan class di app.html, tetapi pendekatan yang lebih "Tailwind" adalah dengan mendefinisikannya di @layer base pada CSS Anda.

Jika Anda ingin mencoba pendekatan ini (opsional, tapi disarankan untuk kebersihan HTML), buka kembali layout.css dan tambahkan ini:

```CSS
@import "tailwindcss";
@custom-variant dark (&:where(.dark, .dark *));

/* Tambahkan Base Layer untuk "Kanvas Utama" */
@layer base {
    body {
        @apply bg-gray-50 text-gray-900 dark:bg-gray-950 dark:text-gray-100;
    }
}
```

Jika Anda melakukan ini, Anda bisa menghapus class bg-gray-50 text-gray-900 ... dari tag <body> di app.html Anda. Ini akan menjaga HTML tetap bersih, dan kanvas utama akan selalu diatur secara otomatis oleh CSS.

############################################################

Prinsip 4: Komponen Reusable (Atomik)

Saat Anda membuat komponen seperti tombol, kartu, atau input, definisikan gaya dark mode-nya langsung di dalam komponen tersebut.

Misalnya, komponen <Card>:

```Svelte
<div class="rounded-xl border border-gray-200 bg-white p-6 shadow-sm dark:border-gray-800 dark:bg-gray-900">
    {@render children()}
</div>
```

3. Sparring Intelektual: Bagaimana "Membatalkan" Secara Nyata?
Tombol "Batalkan" di atas saat ini baru membersihkan UI. Namun, di balik layar, request API (REST/gRPC) atau Navigasi SvelteKit mungkin masih berjalan. Agar boilerplate Anda benar-benar tangguh, Anda perlu memahami konsep AbortController.

Untuk Request API:
Anda harus mengirimkan AbortSignal ke fungsi fetch atau gRPC Anda.

// Contoh implementasi di logic Anda
const controller = new AbortController();

// Di LoadingStore.cancelAll()
controller.abort(); 

// Di API Call
const response = await fetch('/api/data', { signal: controller.signal });
